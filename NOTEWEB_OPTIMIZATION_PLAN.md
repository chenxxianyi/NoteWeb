# NoteWeb 优化开发方案

## 1. 背景

本方案基于近期排查到的几个用户侧问题整理：

- TXT / DOCX 文档进入阅读后仍显示未读。
- 阅读完成后进度异常显示为 1%，未按真实阅读比例展示。
- 首页“最近批注 / 批注总数”刷新后归零，但实际存在批注。
- “最新动态”里的批注、笔记标签无法进入对应阅读或编辑预览页面。
- 设置页面大量控件无法使用，疑似仅完成了静态界面。

目标是把这些功能从“能显示”推进到“状态可靠、刷新不丢、点击可达、设置生效”。

## 2. 问题归因

### 2.1 阅读状态与阅读进度

当前核心问题是阅读状态和进度更新链路不完整：

- 非 PDF 文档的阅读进度曾按外层页面高度计算，容易得到非常小的比例。
- 短文档在首屏完整展示时，如果没有滚动事件，也可能不会被正常标记为读完。
- 标记已读时如果只发空 PATCH，请求不会真正改变 `read_progress`。
- 前端文档列表和当前阅读文档的状态需要同步更新，否则会出现阅读页已变化、列表仍旧显示未读的问题。

建议统一规则：

- 后端和前端内部使用 `0-1` 的小数保存阅读进度。
- UI 展示时再转换成 `0%-100%`。
- `read_progress >= 1` 显示已读或阅读至 100%。
- 保存进度时只允许合理范围内增长，避免从 100% 被回写成 1% 或更低。

### 2.2 首页批注统计刷新归零

首页统计依赖前端内存中的批注列表，刷新页面后内存会清空。如果首页只加载文档和笔记，没有重新按文档拉取批注，就会出现：

- 实际数据库有批注。
- 最近动态能在某些场景下短暂出现。
- 刷新后批注总数变成 0。

建议首页进入时按所有文档批量拉取批注，或者新增后端聚合接口直接返回统计和最近动态。

### 2.3 最新动态点击无跳转

最新动态展示层缺少可点击行为和目标信息：

- 批注动态需要携带 `documentId`，点击后进入 `/reader/:documentId`。
- 笔记动态需要携带 `noteId`，点击后进入 `/notes?noteId=xxx` 并选中对应笔记。
- 列表项需要具备键盘可访问能力，例如 Enter / Space 触发跳转。

### 2.4 设置页功能不可用

设置页目前大部分是静态或半静态 UI：

- 用户名、邮箱只读取当前登录用户，输入框为只读，没有保存入口。
- 上传头像、修改密码、删除账号按钮没有绑定事件。
- 主题色按钮没有点击逻辑。
- 字体下拉没有绑定状态，也没有应用到全局阅读器。
- 阅读模式开关只影响当前页面本地变量，没有持久化和全局应用。
- 导出数据按钮没有导出逻辑。
- 文档数量显示为占位符。
- AI 配置只保存在 localStorage，没有和真实 AI 服务链路打通。
- 后端目前缺少资料更新、密码修改、头像上传、账号删除、用户设置持久化等接口。

## 3. 优化目标

### 3.1 短期目标

- 阅读状态和阅读进度刷新后仍保持正确。
- 首页批注统计刷新后不归零。
- 最近动态点击后能进入对应阅读器或笔记页面。
- 设置页中可以纯前端落地的功能先可用，例如主题、字体、阅读模式、导出数据、文档数量。
- 暂未具备后端能力的按钮给出明确反馈，避免用户误以为点击失效。

### 3.2 中期目标

- 补齐用户资料、密码、头像、删除账号等后端接口。
- 统一用户设置存储策略，支持跨设备同步。
- 增加首页聚合接口，减少前端多次请求。
- AI 配置从“本地保存”升级为“可实际调用并可验证”。

### 3.3 长期目标

- 建立阅读进度、批注、笔记、设置的稳定回归测试。
- 将首页统计和最近动态抽象成明确的数据模型。
- 将设置页拆成可维护的设置模块，避免继续堆叠静态表单。

## 4. 分阶段开发计划

### 阶段一：当前问题收口

优先级：P0

开发内容：

- 确认 `TXT / DOCX / MD` 阅读进度按正文容器计算。
- 短文档首屏完整展示时自动记为 100%。
- 退出阅读页时保存最后一次进度。
- 文档列表、继续阅读、当前阅读文档同步更新 `read_progress`。
- 首页加载时按文档批量拉取批注，并刷新批注总数和最近动态。
- 最近动态添加点击跳转能力。

验收标准：

- TXT / DOCX 打开阅读后不再停留在“未读”。
- 阅读完成后显示“阅读至100%”。
- 刷新首页后批注总数不丢失。
- 点击批注动态进入对应阅读器。
- 点击笔记动态进入笔记页面并选中对应笔记。

### 阶段二：设置页前端可用化

优先级：P1

开发内容：

- 主题色：
  - 点击主题色立即切换全局主题色。
  - 保存到 localStorage。
  - 应用启动时自动恢复。

- 字体：
  - 下拉框绑定当前字体。
  - 应用到阅读器正文和必要的内容区域。
  - 保存到 localStorage。

- 阅读模式：
  - 开关状态保存到 localStorage。
  - 阅读器根据状态切换纸书质感背景或普通背景。

- 文档数量：
  - 从 `documentStore.documents.length` 计算真实数量。
  - 页面进入时确保文档数据已加载。

- 导出数据：
  - 导出文档元信息、笔记、批注、用户设置为 JSON。
  - 文件名建议：`noteweb-export-YYYY-MM-DD.json`。

- 交互反馈：
  - 未开发的头像、密码、删除账号先弹出提示。
  - 避免按钮无响应。

验收标准：

- 切换主题、字体、阅读模式后刷新仍生效。
- 文档数量不再显示 `—`。
- 点击导出数据会下载 JSON 文件。
- 未完成后端能力的按钮都有清楚提示。

### 阶段三：设置页后端能力补齐

优先级：P2

后端接口建议：

- `PATCH /api/auth/profile`
  - 修改用户名、邮箱。

- `POST /api/auth/change-password`
  - 校验旧密码并保存新密码。

- `POST /api/auth/avatar`
  - 上传头像，返回头像 URL。

- `DELETE /api/auth/account`
  - 删除当前账号及关联数据。

- `GET /api/settings`
  - 获取用户设置。

- `PATCH /api/settings`
  - 保存主题、字体、阅读模式、AI 配置等设置。

数据库建议：

- `users` 表补充 `avatar_url` 字段。
- 新增 `user_settings` 表保存用户偏好。
- AI Key 如需后端保存，必须加密存储；更推荐先保存在用户本地浏览器，避免安全风险扩大。

验收标准：

- 修改用户名、邮箱后刷新仍保持。
- 修改密码后旧密码失效，新密码可登录。
- 头像上传后侧边栏和设置页同步展示。
- 删除账号需二次确认，执行后清理关联文档、笔记、批注。
- 设置项跨浏览器登录后可同步。

### 阶段四：首页聚合接口优化

优先级：P2

接口建议：

- `GET /api/dashboard/summary`

返回内容建议：

```json
{
  "documentCount": 5,
  "annotationCount": 12,
  "noteCount": 3,
  "weeklyReadingSeconds": 3600,
  "continueReading": [],
  "recentActivities": []
}
```

收益：

- 首页不用分别请求文档、笔记、多个文档的批注。
- 刷新后的统计更稳定。
- 后端统一排序和裁剪最近动态，避免前端重复聚合。

验收标准：

- 首页首屏请求数量减少。
- 统计数据和最近动态由同一个接口返回。
- 前端刷新、重新登录、跨浏览器访问时结果一致。

## 5. 技术实施要点

### 5.1 前端状态管理

- `documentStore` 负责文档列表、当前文档、阅读进度。
- `annotationStore` 负责批注列表、批量加载和按时间排序。
- `noteStore` 负责笔记列表和当前笔记选择。
- 设置项可先独立封装为 `settingsStore`，避免 `SettingsView.vue` 继续膨胀。

### 5.2 路由跳转

- 批注动态跳转：`/reader/:documentId`。
- 笔记动态跳转：`/notes?noteId=:noteId`。
- `NotesView` 需要监听 query 变化并选中目标笔记。

### 5.3 数据持久化

- 阅读进度、批注、笔记必须以后端为准。
- 主题、字体、阅读模式短期可用 localStorage。
- 用户资料和密码必须走后端接口。
- AI Key 不建议明文保存在服务端，除非实现加密和权限隔离。

### 5.4 用户反馈

- 每个按钮都必须有响应：
  - 已支持：执行真实功能并提示成功或失败。
  - 暂未支持：提示“该功能正在开发中”。
- 保存类操作需要有 loading / success / error 状态。
- 危险操作必须二次确认。

## 6. 测试与验证策略

遵循当前项目偏好：默认不使用 Playwright，不跑 E2E，不做浏览器自动化截图验证，除非明确要求。

默认验证方式：

- 前端：
  - `npm.cmd run build`
  - 如项目存在类型检查或单元测试脚本，则执行对应脚本。

- 后端：
  - `go test ./...`
  - `go build ./cmd/server`

- 人工核验清单：
  - 上传 TXT / DOCX，进入阅读，返回列表查看状态。
  - 阅读到底后刷新，确认进度保持。
  - 新增批注后刷新首页，确认批注总数和最近动态保持。
  - 点击最新动态中的批注和笔记，确认能进入目标页面。
  - 设置主题、字体、阅读模式后刷新，确认设置保持。
  - 导出数据后检查 JSON 内容结构。

## 7. 风险与注意事项

- 当前项目存在 `frontend` 和 `frontend-b` 两套前端目录，开发前需要确认线上实际使用哪一套。
- 如果后端数据模型中 `read_progress` 当前不是 `0-1`，需要先统一字段语义，避免 UI 百分比换算错误。
- 首页批量拉取所有文档批注在文档量大时可能变慢，中期应改为后端聚合接口。
- 删除账号、修改密码、头像上传属于账号安全能力，必须做鉴权、校验和错误处理。
- AI Key 涉及敏感信息，不应随普通导出数据明文导出。

## 8. 建议执行顺序

1. 先完成阶段一，确保当前用户已经遇到的问题全部闭环。
2. 再完成阶段二，让设置页不再像静态展示页。
3. 然后补齐阶段三的后端账号能力。
4. 最后通过阶段四减少首页请求数量，提高长期稳定性。

## 9. 交付清单

- 阅读状态和阅读进度修复。
- 首页批注统计刷新持久化。
- 最新动态点击跳转。
- 设置页前端功能可用化。
- 设置页后端接口补齐。
- 首页聚合接口。
- 构建、后端测试、人工核验记录。
