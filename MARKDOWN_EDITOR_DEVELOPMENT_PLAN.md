# Markdown 文档编辑功能开发方案

## 1. 背景与目标

当前 Markdown 文档页面已经可以正常展示内容，但整体体验更接近“阅读器”或“Markdown 渲染预览页”，不是理想的文档编辑器。目标是参考 Typora 的核心体验，将该页面升级为一个低干扰、所见即所得、可直接编辑的 Markdown 文档编辑功能。

核心目标：

- 默认呈现为排版良好的文档，而不是源码编辑框。
- 点击正文即可直接编辑，编辑过程中保持接近最终阅读效果。
- Markdown 语法标记只在必要时出现，减少视觉干扰。
- 支持常用 Markdown 块级和行内编辑能力。
- 提供大纲、搜索、字数统计、保存状态等文档级辅助功能。
- 保持界面简洁、安静、适合长文写作和技术文档编辑。

## 2. 设计原则

### 2.1 所见即所得

编辑器不采用传统的“左侧源码、右侧预览”作为默认模式。正文区域应直接显示排版结果，用户输入 Markdown 后实时转换为对应结构。

示例：

- 输入 `# 标题` 后自动变成一级标题。
- 输入 `- ` 后自动进入无序列表。
- 输入 `> ` 后自动进入引用块。
- 输入三个反引号后自动进入代码块。

### 2.2 低干扰编辑

默认状态只显示必要控件。工具栏、格式按钮、块操作入口应尽量上下文出现：

- 未选中文本时，不显示行内格式工具。
- 选中文本时，显示浮动工具条。
- 光标进入表格、图片、代码块时，显示对应块操作。
- 左侧文件树和右侧大纲默认可折叠。

### 2.3 阅读与编辑统一

页面不需要明显区分“阅读模式”和“编辑模式”。默认就是可编辑文档，用户可以像阅读文章一样浏览，也可以随时点击内容开始编辑。

### 2.4 Markdown 仍是最终存储格式

虽然编辑体验是可视化的，但数据保存仍应保持 Markdown 文本，避免破坏现有文档体系和可迁移性。

## 3. 页面布局方案

建议采用三栏可折叠布局：

```text
┌─────────────────────────────────────────────────────────────┐
│ 顶部工具栏：返回 / 文档标题 / 搜索 / 大纲 / 更多 / 保存状态 │
├──────────────┬──────────────────────────────┬───────────────┤
│ 左侧文档区   │ 中间 Markdown 编辑画布        │ 右侧大纲区    │
│ 文件树       │ 所见即所得正文编辑            │ 标题目录      │
│ 最近文档     │ 图片/表格/代码块编辑           │ 文档信息      │
└──────────────┴──────────────────────────────┴───────────────┘
```

默认展示建议：

- 左侧文档区默认收起，仅保留入口按钮。
- 右侧大纲在桌面端可默认展开，在小屏幕默认收起。
- 中间正文最大宽度控制在 `760px - 860px`。
- 顶部工具栏保持轻量，不遮挡正文标题。
- 底部或右下角显示字数、保存状态、缩放比例等状态信息。

## 4. 核心功能设计

### 4.1 编辑画布

编辑画布是整个功能的核心，应具备以下能力：

- 正文区域直接可编辑。
- 支持撤销、重做。
- 支持自动保存和手动保存。
- 支持选区格式化。
- 支持 Markdown 快捷输入。
- 支持粘贴 Markdown 内容并保留结构。
- 支持从外部复制富文本并转为 Markdown 结构。

推荐交互：

- 当前编辑行可以轻微显示 Markdown 源符号。
- 非当前行隐藏大部分 Markdown 标记，只保留最终排版效果。
- 保存中、已保存、保存失败状态需要清晰可见。

### 4.2 顶部工具栏

顶部工具栏应从当前偏阅读器的图标组改为编辑器控制栏。

建议包含：

- 返回
- 文档标题
- 保存状态
- 搜索
- 大纲开关
- 文档树开关
- 更多菜单

更多菜单中包含：

- 源码模式
- 专注模式
- 打字机模式
- 导出
- 主题设置
- 文档信息

### 4.3 浮动选区工具条

当用户选中文本时，在选区附近显示工具条。

建议操作：

- 加粗
- 斜体
- 删除线
- 行内代码
- 链接
- 标题等级
- 引用
- 清除格式

要求：

- 工具条不能遮挡选中文本。
- 键盘用户也能通过快捷键完成同等操作。
- 工具按钮需要有明确的 hover、focus、active 状态。

### 4.4 斜杠菜单

在空行输入 `/` 时弹出插入菜单。

建议菜单项：

- 标题 1
- 标题 2
- 标题 3
- 无序列表
- 有序列表
- 任务列表
- 引用
- 代码块
- 表格
- 图片
- 分割线
- 数学公式
- Mermaid 图表

交互要求：

- 支持键盘上下选择。
- 支持回车确认。
- 支持模糊搜索。
- Esc 关闭菜单。

### 4.5 表格编辑

表格不应要求用户手写 Markdown 表格。

表格能力：

- 插入指定行列数的表格。
- 单元格内直接编辑。
- 添加行、删除行。
- 添加列、删除列。
- 设置列对齐方式。
- 支持 Tab 在单元格之间移动。
- 支持粘贴表格内容。

### 4.6 图片编辑

图片能力：

- 支持拖拽上传。
- 支持粘贴图片。
- 支持从本地或 URL 插入图片。
- 支持设置 alt 文本。
- 支持调整显示宽度。
- 支持点击图片后显示图片工具条。

图片工具条：

- 替换图片
- 设置 alt
- 复制路径
- 删除图片
- 调整尺寸

### 4.7 代码块

代码块能力：

- 支持语言选择。
- 支持语法高亮。
- 支持复制代码。
- 支持行号开关。
- 支持代码块折叠。

建议支持：

- JavaScript
- TypeScript
- Go
- Python
- HTML
- CSS
- JSON
- Bash
- SQL

### 4.8 大纲

右侧大纲根据 Markdown 标题自动生成。

能力：

- 展示 `h1 - h6` 标题层级。
- 点击标题跳转到对应位置。
- 当前滚动位置对应标题高亮。
- 标题拖拽排序可作为后续增强，不放入首期。

### 4.9 搜索

搜索能力：

- 文档内搜索。
- 高亮匹配结果。
- 上一个/下一个匹配。
- 显示匹配数量。
- 后续可扩展替换功能。

### 4.10 状态栏

状态栏建议展示：

- 字数
- 字符数
- 阅读时间
- 保存状态
- 当前模式
- 缩放比例

状态示例：

```text
1240 字 · 约 5 分钟阅读 · 已保存 · Markdown · 100%
```

## 5. 视觉风格方案

### 5.1 整体风格

参考 Typora 的安静写作体验，整体视觉应克制、轻量、文档优先。

建议：

- 减少装饰性卡片。
- 不使用强烈渐变或大面积彩色背景。
- 文档正文像一张自然铺开的编辑画布，而不是嵌在卡片中。
- 工具按钮弱化存在感，但 hover 和 focus 状态要明确。

### 5.2 色彩

建议基础色：

- 页面背景：`#F7F3ED` 或更轻的暖灰色。
- 正文背景：透明或 `#FFFFFF`，避免厚重卡片感。
- 主文本：`#171717`。
- 次级文本：`#66615B`。
- 边框：`#E4DED6`。
- 强调色：`#8B6F47` 或项目现有主色。

注意：

- 不要让页面变成单一棕色或米色主题。
- 文本对比度必须满足可读性。
- 状态色不能只依赖颜色表达，需要配合文本或图标。

### 5.3 字体

建议：

- UI 控件：Inter、system-ui。
- 中文正文：系统中文字体优先，例如 `PingFang SC`、`Microsoft YaHei`、`Noto Sans CJK SC`。
- 可提供“阅读主题”后续增强，例如宋体、衬线、等宽主题。

正文排版建议：

- 正文字号：`16px - 18px`。
- 行高：`1.75 - 1.9`。
- 段落间距：`0.8em - 1.1em`。
- 标题上方间距应大于下方间距。

## 6. 技术实现建议

不建议使用简单的 `textarea + preview` 方案，因为它难以实现 Typora 式的所见即所得体验。

推荐方案：

- 编辑器内核：Milkdown、TipTap 或 ProseMirror。
- Markdown 解析与序列化：remark、unified 或 markdown-it。
- 语法高亮：Shiki 或 Prism。
- 数学公式：KaTeX。
- 图表：Mermaid。
- 快捷键管理：编辑器内核插件或自定义 keymap。

优先推荐：

1. 如果项目已有 React/Vue 编辑器生态，优先选择与现有技术栈匹配的库。
2. 如果需要 Markdown 优先和 Typora 相似体验，优先评估 Milkdown。
3. 如果需要更强的富文本可扩展性，优先评估 TipTap/ProseMirror。

## 7. 数据与保存策略

保存策略：

- 编辑器内部维护结构化文档状态。
- 保存时序列化为 Markdown 文本。
- 加载时将 Markdown 解析为编辑器文档结构。
- 自动保存采用防抖策略，例如 800ms - 1500ms。
- 保存失败时保留本地草稿，避免内容丢失。

需要处理的状态：

- 未修改
- 已修改未保存
- 保存中
- 已保存
- 保存失败
- 离线草稿

离开页面前：

- 如果存在未保存内容，需要提示用户。
- 如果自动保存成功，则不打断用户。

## 8. 快捷键设计

建议支持：

| 快捷键 | 功能 |
| --- | --- |
| Ctrl/Cmd + S | 保存 |
| Ctrl/Cmd + B | 加粗 |
| Ctrl/Cmd + I | 斜体 |
| Ctrl/Cmd + K | 插入链接 |
| Ctrl/Cmd + F | 文档内搜索 |
| Ctrl/Cmd + Z | 撤销 |
| Ctrl/Cmd + Shift + Z | 重做 |
| Ctrl/Cmd + / | 打开命令菜单 |
| Esc | 关闭浮层或菜单 |

## 9. 可访问性要求

必须满足：

- 所有按钮可通过键盘访问。
- Tab 顺序符合视觉顺序。
- 自定义浮层、菜单、弹窗具备合理的 ARIA 属性。
- 保存失败、上传失败等错误需要通过文本提示，不只依赖颜色。
- 焦点状态必须清晰可见。
- 图片需要支持 alt 文本。
- 颜色对比度满足正文阅读要求。

## 10. 分阶段开发计划

### P0：编辑器基础改造

目标：把当前页面从阅读器升级为可直接编辑的 Markdown 文档页。

任务：

- 引入编辑器内核。
- 支持 Markdown 加载和保存。
- 实现正文区域直接编辑。
- 支持标题、段落、列表、引用、代码块、链接、图片。
- 实现自动保存和保存状态。
- 实现基础顶部工具栏。
- 实现右侧大纲。
- 实现文档内搜索。

验收标准：

- 打开 Markdown 文档后可以直接点击正文编辑。
- 编辑后能保存为 Markdown。
- 常见 Markdown 结构保存后不丢失。
- 页面在桌面端和移动端无明显布局错乱。

### P1：Typora 式体验增强

目标：提升编辑效率和沉浸感。

任务：

- 实现浮动选区工具条。
- 实现 `/` 插入菜单。
- 实现表格可视化编辑。
- 实现任务列表可点击勾选。
- 实现图片拖拽上传和粘贴上传。
- 实现状态栏，包括字数、阅读时间、保存状态。
- 实现专注模式。
- 实现打字机模式。

验收标准：

- 用户可以不手写复杂 Markdown 语法完成常见编辑。
- 表格、图片、代码块都有对应上下文操作。
- 长文编辑时工具不干扰正文阅读。

### P2：高级 Markdown 能力

目标：覆盖技术文档和复杂文档场景。

任务：

- 支持 Mermaid 图表。
- 支持 KaTeX 数学公式。
- 支持 Front Matter。
- 支持脚注。
- 支持导出 HTML/PDF/DOCX。
- 支持自定义主题。
- 支持源码模式和所见即所得模式切换。

验收标准：

- 技术文档中的图表、公式、代码块都能正常编辑和预览。
- 导出结果与编辑画布视觉尽量一致。
- 源码模式与可视化模式切换不破坏内容。

## 11. 风险与注意事项

### 11.1 Markdown 与富文本结构互转风险

风险：

- 部分 Markdown 扩展语法可能在解析和保存时丢失。

应对：

- 明确支持的 Markdown 语法范围。
- 对不支持的语法保留为源码块或原始节点。
- 添加 Markdown 序列化回归测试。

### 11.2 编辑器库集成成本

风险：

- ProseMirror/TipTap/Milkdown 插件体系有学习成本。

应对：

- 首期只实现基础块和行内格式。
- 表格、公式、图表放到后续阶段。
- 优先使用成熟插件，不手写核心编辑行为。

### 11.3 自动保存与数据丢失

风险：

- 网络异常或保存失败导致内容丢失。

应对：

- 自动保存失败时保留本地草稿。
- 保存失败必须有明确状态提示。
- 离开页面前检查未保存内容。

### 11.4 移动端编辑体验

风险：

- 富文本编辑器在移动端容易出现键盘遮挡、选区错位等问题。

应对：

- 移动端优先保证基础编辑、保存、搜索和大纲。
- 复杂表格编辑可在移动端降级为简化操作。

## 12. 推荐首期范围

建议第一轮不要一次性追完整 Typora，而是先完成以下闭环：

- 单文档所见即所得编辑。
- Markdown 加载与保存。
- 标题、段落、列表、引用、代码块、链接、图片。
- 顶部极简工具栏。
- 右侧大纲。
- 自动保存状态。
- 基础搜索。

完成这部分后，用户就能明显感受到从“阅读页”变成“文档编辑器”。后续再逐步增强表格、斜杠菜单、浮动工具条、公式和图表。
